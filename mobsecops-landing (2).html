// Intégration CI/CD

Intégration en 3 étapes

01

Configurer le webhook GitHub

Dans les settings de votre repo, ajoutez le webhook pointant vers le controller. La signature HMAC est vérifiée automatiquement à chaque event.

POST https://votre-controller:8000/webhook/github X-Hub-Signature-256: sha256=<hmac> X-GitHub-Event: push
02

Démarrer l'infrastructure Docker

Un seul script initialise les volumes partagés, le réseau Docker et démarre tous les services en parallèle.

./init.sh # Crée apk_storage, repos_storage, pipeline_net docker-compose up # Démarre les 11 services
03

Pousser — tout le reste est automatique

À chaque git push, le pipeline se déclenche, analyse l'APK, génère le rapport IA et notifie Discord en moins de 90 secondes.

git push origin main # → Webhook reçu # → Repo cloné, APK détecté # → 9 scans parallèles lancés # → Rapport IA généré (Groq llama-3.1) # → Notification Discord envoyée # → Push: ✅ AUTORISÉ | 🚫 BLOQUÉ

Trigger manuel disponible pour les tests :

POST /trigger/manual {"repo": "myapp", "branch": "main", "commit": "abc123", "apk_filename": "myapp.apk"}

// Flow complet — de git push à Discord

Chaque push, un rapport.
Zéro intervention manuelle.

Voici exactement ce qui se passe entre un git push et la notification Discord — chaque étape, chaque service, chaque décision.

⚡

1. git push origin main

Le développeur pousse sur GitHub. GitHub émet immédiatement un événement push vers le webhook configuré.

X-GitHub-Event: push X-Hub-Signature-256: sha256=…
🎛️

2. Controller reçoit le webhook

Le Controller (:8010) vérifie la signature HMAC-SHA256 avec le secret partagé. Si invalide → 401 rejeté. Si valide → accepté en 202 Accepted immédiatement.

POST /webhook/github HMAC vérifié
📁

3. Clone du repo + détection APK

Le Controller clone le repo via git clone --depth 1 dans /repos_storage/. Si un .apk est trouvé dans le repo, il est copié dans /apk_storage/.

/repos_storage/myapp /apk_storage/myapp_abc123.apk
⚙️

4. Pipeline lancé en background

La fonction run_pipeline() s'exécute en tâche de fond (BackgroundTask). Le Controller répond immédiatement au webhook GitHub sans attendre les scans.

asyncio BackgroundTask {"status":"accepted"}
🔬

5. 9 scanners en parallèle via asyncio.gather()

Tous les scanners APK et repo sont lancés simultanément. Chaque résultat est mis à jour en temps réel dans pipeline_history dès qu'il arrive — le statut passe de running à partiel.

mobsf :8001 gitleaks :8002 syft :8003 androguard :8004 virustotal :8005 permissions :8006 network :8007 obfuscation :8008 ssl :8009
🧠

6. Analyse IA via Groq (llama-3.1-8b)

Les résultats agrégés (top 5 findings par scanner) sont envoyés à l'API Groq. L'IA génère en JSON : niveau de risque global, résumé en 3 phrases, tickets de remédiation avec effort, release notes, et recommandation de push.

Groq API — llama-3.1-8b-instant risk_level · tickets · push_recommendation
🚫

7. Décision automatique de push

Si le niveau de risque est critical ou high, push_allowed = False est forcé, indépendamment de la recommandation IA. Statut final du pipeline : blocked ou success.

critical/high → push_allowed: false medium/low → push autorisé
💬

8. Notification Discord envoyée

Un embed riche est posté sur le webhook Discord configuré : couleur selon le risque, score moyen, top 5 tickets, résumé IA et release notes. L'équipe est notifiée instantanément.

Discord Webhook 🚫 BLOQUÉ ✅ AUTORISÉ
📊

9. Historique disponible via Admin API

Le pipeline finalisé est stocké dans pipeline_history (100 max en mémoire). Consultable à tout moment via GET /admin/pipelines avec la clé admin.

GET /admin/pipelines X-API-Key: <admin_key>
